## 結論

現行配布版は、以前原文返しになっていた GitHub Actions の長文を正常に圧縮できた。固定品質テスト5件も配布版EXEで全件合格し、今回確認した8種類で原文返しは0件だった。

ただし、意味差分をCodexで確認した結果、長文CSVケースで「先頭数行だけを見て文字コードを決めない」という禁止条件が圧縮後に欠落していた。アプリの自動検証はこの出力を採用しているため、現時点の総合判定は **条件付き合格（改善が必要）** とする。

また、短文では意味を保っていても「APIを呼び。」「トランザクションで処理し。」のような不自然な文末が複数発生した。圧縮率も短文5件の平均文字数比が86.5%で、削減量は小さい。

## テスト方法

1. 配布版の `TrimPrompt-avx2.exe --package-smoke-test` を実行し、デスクトップUIと同じ `/api/prepare-compression`、`/api/prepare-input`、`/api/compress` 経路で固定5件を検証した。
2. 配布版が保存するのは先頭ケースの全文だけなので、同じモデル、配布版設定、プロンプト、前処理、後処理、検証器を使うCLI最終パイプラインで固定5件の全文を再取得した。
3. 起動中の実画面に表示されていた長文CSVケースについて、原文、圧縮文、画面上の文字数比、原文推奨を確認した。
4. 以前原文返しになった GitHub Actions 文と、誤字訂正・数値・エラー識別子を含む Next.js 文を同一モデル・設定で再実行した。
5. Codex側では、対象と操作、固有名詞、数値、ファイル名・識別子、否定・禁止、条件・例外、既存挙動、確認要件、追加された意味を比較した。

## 重要な発見

### 1. 長文CSVケースで禁止条件が欠落

判定: **不合格（意味欠落）**

原文に含まれていた重要条件:

> 先頭数行を見ただけで文字コードを決めてデータを渡せるのは避け

圧縮結果には `UTF-8`、`UTF-8 BOM`、`Shift_JIS` の判定自体は残ったが、「先頭数行だけで決めない」という判定方法の禁止が残らなかった。単なる言い換えではなく、実装方法を制限する条件の欠落である。

このほかの `UNSUPPORTED_ENCODING`、対象ファイル名だけを返す、10MB超を読込前に拒否、`INVALID_FILE_SIZE`、空列を詰めない、既存集計を維持、CSV全文をログへ出さない、ストリーム処理、DB部分登録禁止、画面変更不要は保持されていた。

圧縮結果の末尾では `UNSUPPORTED_ENCODING`、10MB、空行、ログ、成功・失敗件数が重複しており、文字数を増やしている。

### 2. 以前のGitHub Actions原文返しは解消

判定: **合格**

圧縮結果:

```text
GitHub Actions Node.js CI(Node.js 22)の15分以上かかる依存取得を改善。pull_request/mainへのpush workflowでactions/setup-node cache: npmを使いpackage-lock.jsonとpackages/*/package-lock.jsonをcacheキー/dependency-cache対象に。npm ci→npm test→npm run lint→npm run typecheckの順序は維持、実行条件は維持。node_modulesはキャッシュしない。cache復元失敗でも通常どおり続行しCI失敗させない。ログにcache hit/cache miss/lockfileを出す。外部fork pull_requestでsecrets を要求しない、contents: read基本、write権限追加しない。初回と2回目確認。
```

次の条件をすべて保持していた。

- `pull_request` と `main` への `push`
- Node.js 22
- `npm ci`、`npm test`、`npm run lint`、`npm run typecheck` の順序と実行条件
- `actions/setup-node` の `cache: npm`
- ルートと `packages/*/package-lock.json`
- `node_modules` をキャッシュしない
- キャッシュ復元失敗時もCIを継続
- `cache hit`、`cache miss`、使用lockfileのログ
- 外部forkでsecrets不要
- `contents: read` とwrite権限を追加しない条件
- キャッシュなし初回とキャッシュあり2回目の確認

文字数比は65.7%で、意味保持と削減量のバランスは良好だった。

### 3. 訂正語・識別子ケースは保持

判定: **合格（文体に軽微な難あり）**

`custmerId` を `customerId`、`INVALID_CUSTMER` を `INVALID_CUSTOMER` に訂正し、`POST /api/orders`、500の現象、`HTTP 400`、成功レスポンス形式、在庫引当、監査ログ、テスト追加、既存形式を変更しない条件を保持した。

文字数は205→175（85.4%）。末尾の「変えるのはやめる」は命令としてやや不自然だが、意味は失っていない。

## 改善優先度

### 優先度A: 条件欠落の検出

「AだけでBするのは避ける」「Aを根拠にBしない」のような、方法・根拠・範囲を伴う禁止条件を保護対象として抽出する。長文CSVケースを回帰fixtureへ追加し、次を必須マーカーとして評価する。

```text
先頭数行 + だけ + 文字コード決定 + 避ける/禁止/しない
```

### 優先度B: 文末の決定的整形

後処理後に、句点直前が `呼び`、`処理し`、`中断し` などの連用形で終わる文と、最終文が `して`、`で` で終わる文を検出する。意味を変えない定型変換で `呼ぶ`、`処理する`、`中断する`、`示す` などへ整える。

### 優先度B: 意味単位の重複除去

文字列一致だけでなく、エラー識別子、数値条件、禁止対象をキーに同じ条件の繰り返しを除去する。長文CSVケースでは末尾の重複を除くだけで、意味を落とさず圧縮率を改善できる。

## 圧縮時間への影響

上記3点は、LLMの再推論を増やさず、前処理・検証・後処理の決定的なルールとして実装できる。文字列走査と既存の制約抽出へ小規模な判定を追加する方式なら、圧縮時間への影響は実質的に無視できる範囲に抑えられる。

二回目のLLM推論による修復は、時間増加と出力の揺れが大きいため、この問題への第一選択にはしない。

## 総合判定

以前の原文返し問題は対象ケースで解消し、エラー識別子、数値、否定、既存挙動の保持は改善している。一方、自動検証を通過したまま方法制約が1件欠落したため、完成扱いにはまだ早い。

次の受入条件は、長文CSVケースを含む回帰評価で条件欠落0件、短文5件で不自然な連用形終止0件、既存30件評価で原文返し・必須条件欠落0件とするのが妥当である。
