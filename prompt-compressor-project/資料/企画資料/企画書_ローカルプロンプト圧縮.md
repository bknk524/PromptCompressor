# ローカルLLMプロンプトリレー 製品概要 v0.3

## 1. これは何か

ローカルLLMプロンプトリレーは、Codexに渡す前の長い依頼文を、
ノートPCのCPUでも動く軽量ローカルLLMで整理し、
必要な条件を保ったまま、より短く、より扱いやすいプロンプトへ変換するアプリである。

ひとことで言えば、
**「Codexへ送る前に、依頼文をローカルで整える前処理レイヤー」**
である。

## 2. 何をしてくれるのか

このアプリは、ユーザーが書いた長文の依頼から、
Codexが作業するうえで本当に必要な情報を抜き出して整理する。

たとえば、以下のような内容を残しやすくする。

- 何をしたいか
- どのファイルが対象か
- どんなエラーが起きているか
- どう直してほしいか
- やってはいけないことは何か
- どこまで確認してほしいか

逆に、以下のような冗長さは圧縮対象にしやすい。

- 長い前置き
- 重複説明
- 迷いの言い回し
- 直接関係しない背景
- 丁寧すぎる補足

## 3. これによって何ができるか

このアプリを使うことで、次のことができる。

1. Codexに渡す入力トークンを減らせる
2. 長文依頼の中で重要条件が埋もれにくくなる
3. 毎回の依頼整理を手でやる負担を減らせる
4. APIキーを預けずにローカルで前処理できる
5. 圧縮後の内容を確認してから送れる

つまり、**コスト削減** と **依頼品質の安定化** を同時に狙える。

## 4. どんな人に向いているか

最初の対象は、Codexを使って日常的に開発や調査を行う人である。

特に向いているのは、次のような利用者である。

- Codexに長文で依頼しがち
- バグ修正やリファクタ依頼を何度も出す
- ログや制約条件を大量に貼ることが多い
- クラウド前処理サービスに機密を渡したくない
- Web UIより、CLIやローカル連携を好む

## 5. 解決したい課題

Codexへの依頼は、実務ではすぐ長くなる。

たとえば、

- 背景説明
- 現状の仮説
- ログ全文
- 対象ファイル一覧
- 修正してほしい内容
- 制約や禁止事項

が一つの依頼に混ざりやすい。

その結果、

- 入力トークンが増えてコストが上がる
- 重要条件が埋もれる
- Codexへの指示が散らばる
- 自分で毎回整理する手間が大きい

という問題が起きる。

このアプリは、その手前で依頼文を整理することで、これを解決しようとする。

## 6. どう動くか

使い方はシンプルである。

1. ユーザーがCodexに渡したい依頼文を用意する
2. アプリに貼る、CLIで渡す、またはCodex連携から呼び出す
3. ローカルLLMが依頼文を圧縮する
4. 圧縮後プロンプト、保持条件、リスクを表示する
5. ユーザーが確認して、必要ならそのままCodexへ渡す

重要なのは、**自動送信しない** ことにある。
最終的に何を送るかは、常にユーザーが決める。

## 7. このアプリの強み

このアプリの価値は、単なる要約ツールではない点にある。

主な強み:

- **ローカル完結**
  APIキー不要で、前処理を端末内で完結できる

- **Codex向け最適化**
  一般会話ではなく、開発依頼で残すべき情報を重視する

- **短くするだけではない**
  重要条件、対象ファイル、エラー、禁止事項を保つことを優先する

- **圧縮レベルを選べる**
  安全重視から強圧縮まで、目的に応じて切り替えられる

- **送る前に確認できる**
  圧縮後の文だけでなく、保持条件や欠落リスクも見られる

- **軽量モデル前提**
  ノートPCのCPUでも実用しやすい構成を前提にしている

## 8. 他と何が違うか

近いカテゴリには、プロンプト最適化ツール、APIプロキシ、LLMゲートウェイがある。
このアプリはそれらと少し立ち位置が違う。

違いは次の通りである。

- クラウドAPIの中継ではなく、**送信前のローカル前処理**
- 汎用チャット向けではなく、**Codex利用に寄せた圧縮**
- 何でも短くするのではなく、**削らない判断を持つ**
- 自動化優先ではなく、**確認可能性と安心感を優先**

## 9. ユースケース

このアプリは、たとえば次のような場面で役に立つ。

### 9.1 バグ修正依頼

長いログ、仮説、試したことが混ざった依頼から、
対象ファイル、再現条件、エラー文、期待する修正内容を整理して渡せる。

### 9.2 リファクタ依頼

背景説明を圧縮しつつ、
壊してはいけない制約や変更範囲を前に出した形へ整えられる。

### 9.3 ログ解析

ログ全文をそのまま投げるのではなく、
必要な箇所と確認したい論点を中心に再構成できる。

### 9.4 設計相談

長い迷いの文章から、
目的、前提、選択肢、比較してほしい観点を短く整理できる。

## 10. 期待できる効果

このアプリで期待する効果は、次の3つである。

### 10.1 トークン消費の抑制

Codexへ送る入力トークンを減らすことで、
無駄な長文化を抑えやすくなる。

### 10.2 依頼品質の安定化

長文依頼の中で条件が散らばる問題を抑え、
毎回の依頼を一定の品質で整えやすくなる。

### 10.3 作業速度の向上

依頼を自分で手直しする時間を減らし、
Codexへ渡すまでの流れを短くできる。

## 11. 商品として見せたい価値

このアプリを商品として説明するなら、
見せるべき価値は次の順番である。

1. **品質を落とさずに短くできる**
2. **ローカルで安全に使える**
3. **結果としてトークン消費を抑えやすい**

単なる削減率だけではなく、
「品質を保ったまま使えること」が核である。

## 12. 初期導入で見たい指標

初期導入では、細かな金額換算よりも
次のような利用指標を見られればよい。

- 1回あたりの推定トークン削減率
- 圧縮後をそのまま使えた割合
- 原文推奨になった割合
- ユーザーが手修正した割合
- 圧縮から送信判断までにかかった時間

このアプリは、
**品質を落とさずに使い続けられるか**
をまず確かめる形を目指す。

金額換算や詳細なコスト可視化は、必要になった時点で後から追加できればよい。

## 13. なぜ今やるのか

Codexや各種クラウドLLMの利用が増えるほど、
入力トークンの積み上がりと、依頼文の長文化が目立ってくる。

一方で、

- 機密性の懸念
- APIキー管理の不安
- クラウド前処理への抵抗

は依然として強い。

そのため、
**ローカルで、安全に、Codex向け依頼文を整えるレイヤー**
には十分な意味がある。

## 14. 最初の提供形態

最初は広げすぎず、次の形に絞る。

- Windows / macOS 対応
- ノートPC CPUで動く軽量ローカルモデル
- ローカルアプリ
- CLI
- ローカルMCP Server
- Codex向けの圧縮モード

最初から全部入りにせず、
日常利用される導線を先に成立させる。

## 15. 今後の広がり

うまく機能する場合、次のように拡張できる。

1. Codex向け圧縮品質の改善
2. profile比較とモデル差し替えの強化
3. GUIの整備
4. 他のコーディングLLM利用への展開
5. チーム向け運用や共有設定への拡張
6. AI PCメーカーやCPU/NPUベンダーとの連携による開発者向け展開

特に、ローカル実行、安全性、クラウド依存の抑制は、
AI PCを訴求するハードウェア企業とも相性が良い。
この方向は最初の提供形態ではなく、
将来の販路拡張案として保持しておく価値がある。

## 16. まとめ

ローカルLLMプロンプトリレーは、
Codexに渡す依頼文を送信前にローカルで整えることで、
**トークン効率、品質、安心感** を同時に整えようとするアプリである。

長文依頼が当たり前になった開発現場に対して、
「そのまま送る」か「自分で頑張って整える」以外の選択肢を提供する。
