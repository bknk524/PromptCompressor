# ローカルLLMプロンプトリレー 要件定義 v0.8

## 1. プロダクト仮説

Codexに送る前に、ユーザーの自然な依頼文をローカルLLMで「意図を落とさない短い実行プロンプト」に変換する。
ユーザーは長い説明、背景、雑談、試行錯誤をそのまま書けるが、Codexへ渡るのは必要最小限の指示・制約・入力データ・出力形式だけになる。

製品名: TrimPrompt

初期フォーカス:

- まずはCodexでの利用を最優先にする。
- 最初の用途は、コーディング依頼、コード修正依頼、ログ解析依頼、実装方針相談の圧縮。
- 将来的に他のクラウドLLMやWeb UIへ展開する。

## 2. 解決したい課題

- Codexへ渡す入力トークンが増え、利用料金やコンテキスト消費が重くなる。
- ユーザーはプロンプトを短くしようとすると、重要な制約や背景まで削ってしまう。
- 毎回「この部分だけ要約して」「この条件は残して」と手作業で調整するのが面倒。
- 企業・個人の利用では、クラウドへ渡す前にローカルで情報を整理・削減したい。
- Codexにコード、ログ、制約、期待する作業範囲をまとめて渡すと長文化しやすい。

## 3. 目標

- Codexに渡す入力トークンを削減する。
- ユーザーの意図、制約、成功条件、出力形式を保持する。
- 誤圧縮や重要情報の欠落をユーザーが送信前に確認できる。
- Codexで使いやすいよう、CLI・クリップボード・ローカルアプリを中心に連携する。
- ローカルCPUでも動く軽量モデルを前提に、日常的に待てる速度で変換する。
- セキュリティ上の懸念を避けるため、MVPではクラウドLLMのAPIキーを保存・利用しない。

## 4. 非目標

- Codexのコーディング能力そのものをローカルLLMで代替すること。
- すべてのプロンプトを必ず短くすること。短縮すると精度が落ちる場合は、短縮しない判断も正解とする。
- 機密情報を完全に自動判定するDLP製品になること。
- 初期MVPで全クラウドLLMのAPIを公式連携すること。
- 初期MVPでクラウドLLMのAPIキー、組織ID、認証情報を取り扱うこと。
- 初期MVPで広く一般向けブラウザWeb UIをサポートすること。

## 5. ターゲットユーザー

- Codexを使ってコーディングや調査を行う開発者。
- 実装依頼、バグ修正依頼、ログ解析依頼が長文化しがちなユーザー。
- ローカル処理によって、Codexへ渡す情報量を事前に制御したいユーザー。
- CLI中心のワークフローを好むユーザー。

## 6. 主要UX

### 6.1 基本フロー

1. ユーザーが通常通り長めの依頼文を書く。
2. 送信前に圧縮レベルを選ぶ。
3. 「圧縮して送る」を押す。
4. ローカルLLMが依頼文を解析し、選択された圧縮レベルに応じてCodex向けの短いプロンプトに変換する。
5. 画面に以下を表示する。
   - 元プロンプト
   - 圧縮後プロンプト
   - 削減率
   - 選択中の圧縮レベル
   - 保持された重要条件
   - 欠落リスクや曖昧な点
6. ユーザーが「このまま送信」「修正」「元のまま送信」を選ぶ。
7. 選択されたプロンプトがCodexへ、CLI・貼り付け・クリップボード経由で渡される。

### 6.2 CodexデスクトップUX

- Codexへ貼り付ける前に、クリップボード内容を圧縮できる。
- 小さな確認ウィンドウで、元文と圧縮後を比較する。
- 「コピー」「原文へ戻す」「再圧縮」を選べる。
- コードタスク向けに、制約、対象ファイル、期待する変更内容が保持されたかを確認できる。

### 6.3 Codex CLI UX

- ターミナルから直接プロンプトを渡し、ローカルLLMで圧縮できる。
- Codexの入力前に、標準出力またはクリップボードへ圧縮後プロンプトを返す。
- 文字列、標準入力、ファイル入力に対応する。
- 圧縮レベル、圧縮モード、出力形式をコマンドオプションで指定できる。
- 開発者や自動化ユーザー向けに、JSON出力も選べる。

### 6.4 APIを使わないCodex連携UX

- クラウドLLMのAPIキーは取得・保存・送信しない。
- Codexへの入力は、ユーザーの手元でCLIまたは貼り付け操作として完結させる。
- 圧縮後プロンプトはクリップボードまたは標準出力へ返す。
- 将来的なAPIプロキシは、ユーザーが明示的に有効化する上級者向けオプションとして扱い、MVPには含めない。

### 6.5 将来的な汎用LLM連携UX

- 将来的には他のクラウドLLM、Web UI、ブラウザ拡張へ展開できる構成にする。
- ただし初期MVPではCodex利用に絞る。

コマンド例:

```powershell
trimprompt "家計簿アプリを作りたい。収支入力、月別合計、カテゴリ集計が必要"
```

```powershell
Get-Content .\prompt.txt | trimprompt --level 2 --copy
```

```powershell
trimprompt --file .\prompt.txt --mode instruction --level 3 --format json
```

出力例:

```text
目的: 家計簿アプリ作成
機能: 収支入力、月別合計、カテゴリ集計
出力: 画面設計と実装方針
```

## 7. 圧縮の考え方

単純な要約ではなく、LLMに命令するための情報だけを抽出する。

必ず残す要素:

- ユーザーの目的
- 入力データや対象範囲
- 守るべき制約
- 禁止事項
- 出力形式
- 品質基準
- 前提条件
- 例外条件
- 言語、トーン、粒度
- 参照すべき固有名詞や数値

削ってよい可能性が高い要素:

- 挨拶
- 重複した説明
- 迷いの文章
- LLMに不要な背景雑談
- 既に制約として抽出済みの長い言い換え
- 結論に影響しないメタコメント

削ってはいけない可能性が高い要素:

- 法務、医療、金融、セキュリティなど高リスク領域の条件
- コード、データ、引用、ログ、エラーメッセージ
- 数値条件
- 否定条件
- 出力フォーマット指定
- 「必ず」「除外」「のみ」「しないで」などの制約語

## 8. 圧縮例

### 8.1 通常タスクの例

元プロンプト:

```text
今度うちのチームで新しい社内ナレッジ共有ツールを導入しようと思っています。
ただ、みんな忙しくて長い説明を読む文化があまりないので、できるだけ短くてわかりやすい導入案内文にしたいです。
対象はエンジニア、営業、CSが混ざった30人くらいの組織です。
Notionみたいなものを使う予定ですが、具体的なツール名はまだ出したくありません。
堅苦しすぎると読まれないので、少しカジュアルだけど仕事っぽさは残してください。
Slackに投稿する想定で、最初に一文でメリット、そのあと箇条書きで使い方、最後に協力依頼を入れてください。
長さは500字以内がいいです。
```

圧縮後プロンプト:

```text
Slack投稿用の社内ナレッジ共有ツール導入案内文を作成してください。
対象はエンジニア、営業、CSを含む約30人の組織です。
ツール名は出さないでください。
トーンは少しカジュアルだが業務向けにしてください。
構成は、最初に一文でメリット、次に箇条書きで使い方、最後に協力依頼。
500字以内。
```

保持された条件:

- 対象読者
- Slack投稿
- ツール名を出さない
- トーン
- 構成
- 500字以内

削除された内容:

- 導入検討中の背景
- 「忙しくて読まれない」という補足
- Notionの例示

### 8.2 開発タスクの例

元プロンプト:

```text
Reactで作っている管理画面の一覧ページがあって、検索フォームのところを直したいです。
今はキーワード入力とステータス選択があるんですが、検索ボタンを押したときだけ検索する仕様にしたいです。
現状は入力するたびに検索が走ってしまってAPIが無駄に呼ばれています。
ただし、ページングのページ番号を変えたときは今まで通り即検索で大丈夫です。
URLクエリにも検索条件を残していて、そこは維持したいです。
TypeScriptで、既存のuseSearchParamsを使っています。
できれば大きなリファクタはしないで、最小変更にしてください。
```

圧縮後プロンプト:

```text
React + TypeScriptの一覧ページを修正してください。
検索条件はキーワード入力とステータス選択です。
入力変更時にはAPI検索を実行せず、検索ボタン押下時だけ検索してください。
ページ番号変更時は従来通り即検索してください。
URLクエリへの検索条件保持は維持してください。
既存のuseSearchParamsを使い、大きなリファクタは避けて最小変更にしてください。
```

保持された条件:

- React + TypeScript
- 入力変更では検索しない
- 検索ボタン押下で検索
- ページング変更では即検索
- URLクエリ維持
- useSearchParams利用
- 最小変更

削除された内容:

- APIが無駄に呼ばれるという背景説明
- 「管理画面」という、実装指示に直結しない補足

### 8.3 圧縮しすぎない例

元プロンプト:

```text
以下の契約条項について、当社に不利な点を洗い出してください。
ただしこれは法的助言ではなく、社内レビュー前の論点整理として使います。
条項の文言は一字一句変えずに引用してください。
リスクは高・中・低で分類してください。
不明点は推測せず、確認質問として分けてください。
日本法を前提にしないでください。準拠法はまだ未定です。
出力は表形式にしてください。

第12条 ...
第13条 ...
```

圧縮後プロンプト:

```text
以下の契約条項について、社内レビュー前の論点整理として、当社に不利な点を洗い出してください。
これは法的助言ではありません。
条項の文言は一字一句変えずに引用してください。
リスクは高・中・低で分類してください。
不明点は推測せず、確認質問として分けてください。
日本法を前提にしないでください。準拠法は未定です。
出力は表形式にしてください。

第12条 ...
第13条 ...
```

この例では、契約条項本文・否定条件・前提条件・出力形式が重要なため、大きく短縮しない。
アプリは削減率が低くても、重要条件を守る方を優先する。

## 10. 非機能要件

### 10.1 性能

- 一般的なノートPCのCPUで動作する。
- 短いプロンプトでは数秒以内、長いプロンプトでも待機が許容できる時間で返す。
- ローカルLLMの処理が遅い場合でも、ユーザーは元のプロンプトをそのまま送れる。
- UIは圧縮処理中に固まらない。

### 10.2 精度

- 重要条件を黙って削除しない。
- 確信が低い場合は、圧縮せず元文の該当部分を残す。
- 圧縮後プロンプトには、曖昧さを増やす表現を入れない。
- 数値、否定条件、出力形式は特に保守的に扱う。

### 10.3 プライバシー

- 元プロンプトは原則ローカル処理する。
- 圧縮前の全文をクラウドへ送らない設計を基本とする。
- 履歴保存はユーザーがONにした場合のみ。
- ローカル履歴は削除できる。
- MVPではクラウドLLMのAPIキー、組織ID、認証情報を保存しない。
- 将来的にAPI連携を追加する場合も、デフォルトOFFにし、OSの安全な資格情報ストアに保存する。WindowsではCredential Manager、macOSではKeychainを想定する。

### 10.4 互換性

- 初期はCLIとローカルアプリを主軸にする。
- デスクトップアプリはWindowsとmacOSを初期対象にする。
- Windows/macOSで同じ中核機能を提供し、OS差分はショートカット、常駐起動、通知、クリップボード連携の層に閉じ込める。
- 将来的にLinuxも対象にできる構成にする。
- 将来的に他のクラウドLLMへ展開できる抽象化を持つ。

## 11. MVP案

最初のMVPは「ローカルアプリ + CLI + クリップボード連携」がよい。

MVPに含める:

- Windows/macOSローカルアプリ
- CLIコマンド
- 軽量ローカルLLM実行部分
- 圧縮後プロンプト生成
- 差分確認画面
- トークン削減率
- 重要条件チェック
- Codex向けのコピー導線
- 元文のまま送る導線
- コードタスク向け保持チェック

MVPから外す:

- 複数人チーム管理
- 高度なDLP
- 全LLMサービスの公式API連携
- クラウドLLM APIキー管理
- APIプロキシ
- ブラウザ拡張
- 自動送信
- 長期履歴分析

## 13. 圧縮モード

- Lossless: できるだけ意味を変えず、重複だけ削る。
- Instruction Extract: 実行指示、制約、出力形式だけを抽出する。
- Codex Optimized: Codexで使いやすい命令形式へ整える。
- Privacy Redaction: クラウド不要な個人情報や内部情報を伏せる。
- Developer Mode: コード、ログ、エラー、再現手順を保守的に保持する。

## 14. 圧縮レベル

精度と圧縮率はトレードオフになりやすいため、ユーザーが圧縮強度を選べるようにする。
圧縮モードは「何を重視して圧縮するか」、圧縮レベルは「どれくらい強く圧縮するか」を表す。

| レベル | 名称 | 方針 | 想定用途 | 欠落リスク |
| --- | --- | --- | --- | --- |
| 0 | Original | 圧縮せず元文を使う | 法務、医療、金融、契約、コード原文が重要な場合 | なし |
| 1 | Safe | 重複、挨拶、迷いの文章だけを削る | 精度を最優先したい通常利用 | 低 |
| 2 | Balanced | 目的、制約、入力、出力形式を残し、背景を整理する | デフォルト設定 | 中 |
| 3 | Aggressive | 実行に必要な指示だけへ強く圧縮する | コスト削減を優先する日常タスク | 中〜高 |
| 4 | Minimal | ほぼ命令テンプレートだけにする | 失敗しても再試行しやすい軽いタスク | 高 |

### 14.1 レベル別の挙動

Level 0 Original:

- 元プロンプトをそのまま送る。
- アプリはトークン数だけ表示する。
- 高リスク領域や原文保持指定がある場合の安全な選択肢。

Level 1 Safe:

- 挨拶、重複、言い直し、不要な前置きだけを削る。
- 例文、数値、否定条件、出力形式、固有名詞は原則保持する。
- 削減率より意味保持を優先する。

Level 2 Balanced:

- デフォルトの推奨レベル。
- 背景を要約し、目的・制約・出力形式・禁止事項を明示する。
- 一般的な文章作成、調査、開発相談に向く。

Level 3 Aggressive:

- 背景情報を大きく削り、実行指示中心にする。
- 例示は意図だけに置き換える。
- 送信前に欠落リスクの確認を強める。

Level 4 Minimal:

- 最短の命令セットに近づける。
- トーンや細かいニュアンスは失われる可能性がある。
- ユーザーに「精度低下の可能性が高い」ことを明示する。

### 14.2 自動推奨

アプリは入力内容を見て推奨レベルを提示する。
ただし、最終選択はユーザーに残す。

推奨例:

- 契約、規約、医療、金融、セキュリティ: Level 0〜1
- コード修正、ログ解析、仕様確認: Level 1〜2
- 文章作成、メール作成、要約: Level 2〜3
- 雑な壁打ち、軽いアイデア出し: Level 3〜4

### 14.3 UI表示

圧縮結果画面では、レベルごとに以下を表示する。

- 推定トークン数
- 推定削減率
- 欠落リスク
- 保持された重要条件
- 削除された内容の要約
- 原文送信への戻しボタン

ユーザーがレベルを切り替えると、圧縮後プロンプトと削減率が再計算される。

## 15. ローカルLLMへの内部プロンプト方針

ローカルLLMには、圧縮後プロンプトだけでなくメタ情報も返させる。
ただしCodexへ送るのは圧縮後プロンプトのみ。

## 21. トークン削減に役立つ手法

トークン削減は、単純な要約だけで行うと精度低下につながりやすい。
本アプリでは、情報の役割ごとに削る・残す・置き換える判断をする。

### 21.1 重複除去

同じ意味の説明、言い換え、前置きを削除する。

例:

```text
長くて読みにくいので、簡潔にしてください。できるだけ短めでお願いします。
```

圧縮:

```text
簡潔にしてください。
```

効果:

- 日常的な依頼で最も安全に削減できる。
- 精度低下が起きにくい。

### 21.2 命令・制約・入力の分離

長文の中から、LLM実行に必要な要素を分けて再構成する。

抽出する要素:

- 目的
- 入力
- 制約
- 禁止事項
- 出力形式
- 評価基準

圧縮後は文章ではなく、短い箇条書きの実行指示にする。

例:

```text
目的: 採用候補者向けメールを作成
条件: 丁寧、200字以内、面接日程の候補を3つ提示
禁止: 過度にカジュアルにしない
```

### 21.3 背景情報の役割判定

背景をすべて消すのではなく、回答に影響する背景だけを残す。

残す背景:

- 対象読者
- 業界やドメイン
- 意思決定に必要な前提
- トーンに影響する事情

削れる背景:

- 依頼に至った経緯
- 迷いの文章
- 作業者の感想
- 結果に影響しない補足

### 21.4 例示の圧縮

長い例をそのまま送らず、例の意図だけを短くする。

例:

```text
この文章みたいに、やさしいけど幼すぎない感じで、少しだけ親しみがあるトーンにしてください。
```

圧縮:

```text
トーン: やさしく、幼すぎず、少し親しみがある。
```

注意:

- 文体模倣が重要な場合は、例文を削りすぎない。
- ブランド文体、法的文言、引用文は原文保持を優先する。

### 21.5 出力形式のテンプレート化

毎回長く書かれる出力形式をテンプレートIDに置き換える。
ただし、クラウドLLM側がテンプレートIDを理解できない場合は、送信時に短い定型文へ展開する。

例:

```text
出力形式: risk_table_v1
```

送信時展開:

```text
表形式で、項目、リスク、理由、対応案を出力してください。
```

効果:

- ユーザー入力側の再利用性が上がる。
- アプリ内でよく使う形式を短く管理できる。

### 21.6 固有名詞・長い語句の一時ラベル化

長いプロジェクト名、部署名、製品名が何度も出る場合、短いラベルに置き換える。

例:

```text
次世代顧客データ統合プラットフォーム移行プロジェクト
```

圧縮:

```text
本プロジェクト
```

注意:

- 複数の固有名詞がある場合は混同リスクがあるため、ラベル表を残す。
- 公式文書や契約文では勝手に置換しない。

### 21.7 コード・ログ・データの選択送信

コード、ログ、CSV、JSONはトークンを大量に使う。
全文送信ではなく、タスクに必要な範囲だけを選ぶ。

削減方法:

- エラーメッセージ周辺だけを残す。
- 関連関数だけを残す。
- CSVは列名、代表行、異常行だけを残す。
- JSONは必要なキーだけを残す。
- 長いログは時刻、エラー種別、スタックトレースの先頭と末尾を残す。

注意:

- 再現に必要なコードは削らない。
- 行番号、ファイル名、エラーコードは保持する。

### 21.8 会話履歴の要約メモリ化

クラウドLLMへ毎回すべての会話履歴を送らず、ローカルで短い作業メモリに変換する。

保持する内容:

- 決定事項
- 未解決の論点
- ユーザーの好み
- 現在のタスク状態
- 参照中のファイルやデータ

削れる内容:

- 過去の試行錯誤
- 解決済みの枝葉
- 雑談
- 同じ説明の繰り返し

### 21.9 RAG・検索による必要文脈だけの投入

長い資料全体を送る代わりに、ローカル検索で関連箇所だけを取り出す。

流れ:

1. ユーザー依頼をローカルで解析する。
2. ローカル資料から関連チャンクを検索する。
3. 上位の必要部分だけをクラウドLLMへ渡す。
4. 参照元を短く添える。

効果:

- 長文資料、議事録、仕様書、ログ分析で効きやすい。
- 情報を削るよりも、必要箇所だけ選ぶ方が精度を保ちやすい。

### 21.10 モデル別トークナイズ最適化

同じ文字数でも、モデルや言語によってトークン数は変わる。
送信先モデルごとに推定トークン数を計算し、削減効果を表示する。

方針:

- 日本語は無理に英語化しない。
- JSONは便利だが冗長になりやすいため、必要な場合だけ使う。
- 箇条書きは多くのタスクで読みやすさと短さのバランスがよい。
- 過度な略語は誤解を増やすため避ける。

### 21.11 プロンプトキャッシュ活用

一部のクラウドLLMでは、同じ長いprefixを繰り返し送る場合にキャッシュでコストや遅延を下げられることがある。
これは入力トークンそのものを減らす手法ではないが、実利用コスト削減には効く可能性がある。

向いているもの:

- 固定のシステムプロンプト
- 長い社内ルール
- 変更頻度の低い出力仕様
- 毎回同じテンプレート

注意:

- キャッシュ仕様はプロバイダー依存。
- Web UI利用では制御しにくい。
- MVPでは削減率とは別指標として扱う。

### 21.12 圧縮しない判断

トークン削減より精度維持を優先すべき場合がある。

圧縮しない、または弱く圧縮する条件:

- 法務、医療、金融、セキュリティなど高リスク領域
- 契約条項、引用、仕様文、コードの原文が重要
- 否定条件が多い
- 数値条件が多い
- ユーザーが「原文を維持」と指定している
- 圧縮後に意味の欠落リスクが高い

この場合、アプリは「削減率は低いが安全」と表示する。

## 22. ローカルLLMモデル選定

### 22.1 前提

本プロダクトでローカルLLMに求める役割は、Codexの代替ではない。
役割は以下に限定する。

- 長い依頼文から目的、制約、対象ファイル、エラー、期待する変更内容を抽出する
- 重複や背景雑談を削る
- 圧縮後の欠落リスクを示す
- 必要に応じて構造化JSONを返す

つまり、重いコード生成モデルをローカルで回すよりも、
小さめで指示追従が強く、コードと日本語をある程度理解できるモデルが向いている。

### 22.2 選定基準

- CPUローカル実行しやすいこと
- まずは 0.6B〜1.7B 級を主軸に検討すること
- 3B〜4B級は品質比較用または上位オプションとして扱うこと
- 日本語入力を扱いやすいこと
- コード、ログ、エラー、ファイル名を壊しにくいこと
- 命令抽出、要約、再構成に強いこと
- CLIやローカルランタイムで扱いやすいこと
- 商用利用や配布を想定したライセンス確認がしやすいこと

### 22.3 候補モデル

#### A. Qwen3-1.7B

位置づけ:

- 第一候補
- デフォルト候補

向いている理由:

- 1.7B級で比較的軽く、コーディング、推論、多言語、指示追従のバランスが良い
- thinking mode と non-thinking mode を切り替えられる
- ローカル利用で llama.cpp、LM Studio、Ollama などのサポートが進んでいる
- Apache 2.0 ライセンス
- 100以上の言語をサポートし、32Kコンテキストを持つ

この用途での使い方:

- 圧縮用途では non-thinking mode を基本にする
- 長い推論をさせず、短い構造化出力を返させる

懸念:

- thinking mode を有効にすると、圧縮器としては遅くなりやすい
- それでも超軽量ではないため、弱いCPUではさらに軽い fallback が必要

#### B. Qwen2.5-Coder-1.5B-Instruct

位置づけ:

- コードタスク向けの主力候補
- 第二候補

向いている理由:

- 1.5B級で比較的軽い
- code generation、code reasoning、code fixing を強く意識した系列
- 32Kコンテキスト
- Apache 2.0 ライセンス

この用途での使い方:

- バグ修正依頼、ログ解析依頼、差分指示の圧縮に向く
- Developer Mode の主候補として相性がよい

懸念:

- 汎用対話や日本語の自然な書き換えでは、Qwen3-1.7Bより不利な可能性がある
- コード圧縮には強いが、全体デフォルトにはやや尖っている

#### C. Gemma 3 1B IT

位置づけ:

- 超軽量の多言語候補
- 第三候補

向いている理由:

- 1B級で非常に軽い
- 32K入力コンテキスト
- 140以上の言語を含む学習データ
- ラップトップやデスクトップでの利用を明示している

この用途での使い方:

- Safe モード中心の軽量圧縮器として向く
- 低スペックCPU向けの fallback 候補

懸念:

- 1B級なので、複雑なコード依頼では保持精度が落ちる可能性がある
- Gemma ライセンスは Apache/MIT より確認項目が多い

#### D. Qwen3-0.6B

位置づけ:

- 最軽量候補
- 第四候補

向いている理由:

- 0.6B級でかなり軽い
- Qwen3系列なので、thinking/non-thinking、100以上の言語、ローカル対応の恩恵を持つ
- 32Kコンテキスト
- Apache 2.0 ライセンス

この用途での使い方:

- Safe モード専用の超軽量 fallback として使う
- クリップボード圧縮の高速モードに向く

懸念:

- モデルが小さいため、Aggressive / Minimal では重要条件の取りこぼしが増えやすい
- コードタスクの精度は 1.5B〜1.7B 級より落ちる可能性が高い

#### E. Phi-4-mini-instruct

位置づけ:

- 品質比較用の上位候補
- 第五候補

向いている理由:

- 3.8B級で、128Kコンテキスト、24言語、日本語対応
- memory/compute constrained environments と latency bound scenarios を想定している
- reasoning と instruction adherence が強い
- MIT ライセンス

この用途での使い方:

- 欠落リスク判定や、厳密な条件保持チェックに向く
- 軽さより品質を優先する比較対象として使う

懸念:

- 軽量優先の主力としては重い
- reasoning寄りで、単純圧縮にはやや過剰な場合がある

#### F. Qwen3-4B

位置づけ:

- 品質比較用の上位候補
- 第六候補

向いている理由:

- 4B級で、コーディング、推論、多言語、指示追従のバランスが良い
- thinking mode と non-thinking mode を切り替えられる
- ローカル利用で llama.cpp、LM Studio、Ollama などのサポートが進んでいる
- Apache 2.0 ライセンス

この用途での使い方:

- 評価ベンチでの上限品質確認用
- 上位マシン向けオプションモデル

懸念:

- ノートPCのCPU運用では主力として重い可能性がある
- 最初の標準モデルにはしない方が無難

#### G. Llama 3.2 1B / 3B Instruct

位置づけ:

- 超軽量 fallback 候補
- 端末相性確認用候補

向いている理由:

- 1B と 3B があり、on-device use cases with limited compute resources を明示している
- query and prompt rewriting が intended use に含まれる

この用途での使い方:

- 弱い端末向け fallback
- Safe モード中心の軽い圧縮器として使う

懸念:

- 現時点では Qwen3 や Phi-4-mini より、コーディング圧縮の主力候補としては弱い
- ライセンスは Apache/MIT ではない

### 22.4 初期推奨

最初のMVPでは、1モデル固定ではなく 2〜4モデルを並行評価する。

推奨順位:

1. Qwen3-1.7B
2. Qwen2.5-Coder-1.5B-Instruct

4. Qwen3-0.6B

おすすめ構成:

- 生成長は短く制限する
- JSON出力ではキーを固定する

```json
  "risk_flags": ["..."],
  "should_send_original": false

```

## 23. モデル差し替えを前提にした設計

### 23.1 方針

モデル選定は初期段階で確定しない前提にする。
そのため、アプリ全体を特定モデルに密結合させず、モデル差し替え、比較、A/B評価をしやすい構成にする。

### 23.2 設計原則

- 圧縮ロジックとモデル実行ロジックを分離する
- プロンプトテンプレートとモデル設定を分離する
- モデル固有の前処理、後処理、停止条件を adapter 層に閉じ込める
- 返却形式を共通JSONに統一する
- 複数モデルを同一評価セットで比較できるようにする
- デフォルトモデルを変更しても UI や CLI の利用方法が変わらないようにする

### 23.3 推奨アーキテクチャ

```text
UI / CLI
  ↓
Compression Service
  ↓
Compression Policy
  ↓
Model Adapter Interface
  ├─ Qwen3Adapter
  ├─ QwenCoderAdapter
  ├─ GemmaAdapter
  ├─ PhiAdapter
  └─ LlamaAdapter
  ↓
Runtime Backend
  ├─ llama.cpp
  ├─ Ollama
  ├─ LM Studio
  └─ Transformers
```

### 23.4 分離すべき責務

#### UI / CLI

- 入力受付
- 圧縮レベル選択
- 結果表示
- コピー、保存、比較

#### Compression Service

- 共通入力形式の受け取り
- 圧縮フロー実行
- モデル adapter 呼び出し
- 結果の共通形式化

#### Compression Policy

- どの要素を残すかのルール
- task type ごとの重みづけ
- Safe / Balanced / Aggressive の振る舞い
- fallback 条件

#### Model Adapter

- モデルごとの prompt template
- thinking on/off 設定
- stop token
- max token
- JSON出力の安定化
- モデルごとの注意点吸収

#### Runtime Backend

- 実際の推論実行
- モデルロード
- 量子化モデル選択
- CPU/GPUの実行切り替え

### 23.5 共通インターフェース

入力インターフェース例:

```json
{
  "task_type": "coding",
  "compression_mode": "codex_optimized",
  "compression_level": 2,
  "input_text": "...",
  "constraints": {
    "preserve_code_blocks": true,
    "preserve_file_names": true,
    "preserve_error_messages": true
  }
}
```

出力インターフェース例:

```json
{
  "model_id": "qwen3-1.7b",
  "runtime": "llama.cpp",
  "distilled_prompt": "...",
  "preserved_requirements": ["..."],
  "removed_content_summary": ["..."],
  "risk_flags": ["..."],
  "should_send_original": false,
  "metrics": {
    "input_tokens_est": 1200,
    "output_tokens_est": 420,
    "compression_ratio": 0.35,
    "latency_ms": 1800
  }
}
```

### 23.6 モデル設定の外出し

モデルごとの設定はコードに埋め込まず、設定ファイルで切り替えられるようにする。

設定項目例:

- model_id
- runtime backend
- quantization
- context length
- default temperature
- default max output
- thinking enabled
- stop sequences
- json mode prompt
- task-specific prompt template

例:

```yaml
models:
  qwen3_1_7b:
    runtime: llama.cpp
    model_path: local-model-files/qwen3-1.7b-q4.gguf
    thinking: false
    default_max_output: 256
    prompt_template: general-codex-compression-v1

  qwen2_5_coder_1_5b:
    runtime: llama.cpp
    model_path: local-model-files/qwen2.5-coder-1.5b-q4.gguf
    thinking: false
    default_max_output: 256
    prompt_template: codex_dev_compress_v1
```

### 23.7 モデル切り替えUX

ユーザーはモデルを直接選べてもよいが、初期UXでは難しくしすぎない。

推奨:

- 通常表示では「標準」「コード重視」「軽量」「高精度」のような抽象名を見せる
- 詳細設定で実際のモデル名を見せる
- ベンチ比較画面でモデル別結果を見られるようにする

例:

- 標準: Qwen3-1.7B
- コード重視: Qwen2.5-Coder-1.5B-Instruct
- 軽量: Qwen3-0.6B
- 高精度: Phi-4-mini-instruct

### 23.8 比較しやすい仕組み

モデル選定をしやすくするため、同一入力に対して複数モデルの出力を並べて比較できるようにする。

比較項目:

- 圧縮後プロンプト
- 推定削減率
- 欠落リスク
- 重要条件保持
- 応答時間
- JSON整形安定性

比較モード:

- 手動比較: ユーザーが2モデルを選ぶ
- 自動比較: デフォルトモデルと候補モデルを裏で比較する
- 回帰テスト: 保存済み評価セットで一括比較する

### 23.9 fallback 設計

モデル差し替えが容易でも、実行失敗時の挙動が一貫していないと運用しづらい。
そのため fallback を設計に含める。

fallback 例:

1. Qwen3-1.7B がタイムアウト
2. Qwen3-0.6B で Safe モード再試行
3. それでも不安定なら元文を返す

### 23.10 初期実装のおすすめ

最初から全 runtime をサポートしない。
まずは一番実験しやすい 1 backend に絞る。

おすすめ順:

1. llama.cpp
2. Ollama
3. Transformers
4. LM Studio

理由:

- llama.cpp は CPU ローカル運用との相性がよい
- GGUF 量子化モデルが豊富
- モデル差し替えがしやすい

### 23.11 次に決めること

- 初期 backend を llama.cpp に固定するか
- モデル設定ファイルを YAML にするか JSON にするか
- 比較モードを MVP に含めるか
- adapter 層の最小責務をどこまでにするか

### 23.12 モデル変更容易性を担保する必須要件

モデル選定をしやすくするには、単に adapter を用意するだけでは足りない。
アプリ全体として「モデル変更がコード変更を伴わない」状態を目標にする。

必須要件:

- UI と CLI から特定モデル名を直接参照しない
- Compression Service 内で `if model == ...` のような分岐を持たない
- デフォルトモデル変更は設定ファイルの差し替えだけで完了する
- モデル変更時に変わる可能性がある要素をすべて外出しする
- モデル比較、昇格、ロールバックを運用手順として定義する

外出し対象:

- model id
- model path
- runtime backend
- quantization
- prompt template
- thinking on/off
- tokenizer
- timeout
- retry policy
- fallback model
- capability metadata

### 23.13 プロファイル別名で選ぶ設計

ユーザーが毎回生のモデル名を選ぶ設計にすると、比較はしやすくても日常利用の UX が重くなる。
そのため、ユーザーは原則として profile を選び、profile が内部でモデル参照先を持つ。

例:

```yaml
profiles:
  standard:
    label: 標準
    model_ref: qwen3_1_7b
    compression_policy: balanced
    fallback_profile: lightweight

  code_focused:
    label: コード重視
    model_ref: qwen2_5_coder_1_5b
    compression_policy: codex_dev
    fallback_profile: lightweight

  lightweight:
    label: 軽量
    model_ref: qwen3_0_6b
    compression_policy: safe
    fallback_profile: original
```

この構成にすると、標準モデルを切り替える作業は `standard.model_ref` の変更で済む。

### 23.14 モデル能力をメタデータで管理する

モデル選定をしやすくするには、単なる名前の一覧ではなく、比較可能な能力情報を持たせる。
この情報はコードに埋め込まず、設定で管理する。

管理項目例:

- code strength
- Japanese strength
- JSON stability
- CPU latency tier
- minimum RAM estimate
- recommended use cases
- supports thinking toggle
- supports long context
- preferred quantization

例:

```yaml
models:
  qwen3_1_7b:
    family: qwen3
    runtime: llama.cpp
    tokenizer: qwen3
    capabilities:
      code_strength: medium
      japanese_strength: medium
      json_stability: medium
      cpu_latency_tier: fast
      min_ram_gb: 4
      recommended_for:
        - codex_general
        - prompt_compression
```

これにより、UI では「速い順」「コード向き」「日本語向き」で並べ替えられる。

### 23.15 選定しやすい評価・昇格フロー

モデル変更を容易にしても、評価フローがなければ選定は感覚的になりやすい。
そのため、候補モデルを登録してから昇格するまでの流れを固定する。

推奨フロー:

1. 候補モデルを `model-catalog.yaml` に追加する
2. 既存 profile とは別に比較用 profile を作る
3. 保存済み評価セットで回帰比較する
4. 基準を満たした場合のみ既存 profile の `model_ref` を差し替える
5. 問題があれば直前の `model_ref` に戻す

最低限見る指標:

- 重要条件保持率
- 圧縮率
- 平均応答時間
- p95 応答時間
- JSON parse 成功率
- fallback 発生率

「モデルを変える」と「本番の標準 profile を変える」を分離するのが重要である。

### 23.16 推奨ディレクトリ構成

モデル関連の変更箇所を明確にするため、設定、adapter、runtime、評価セットの置き場所を分ける。

例:

```text
settings/
  model-catalog.yaml
  compression-profiles.yaml
  runtime-backends.yaml

src/
  services/
    compression-service.ts
  compression-policies/
    balanced-policy.ts
    codex-policy.ts
  adapters/
    model-adapter.ts
    qwen-adapter.ts
    gemma-adapter.ts
  runtimes/
    runtime-backend.ts
    llama-cpp-backend.ts
    ollama-backend.ts

evals/
  codex-core-v1.jsonl
  codex-bugfix-v1.jsonl
  codex-refactor-v1.jsonl
```

この構成なら、モデル差し替え時に触る場所が限定される。

### 23.17 MVP でまず入れるべき変更容易性

MVP の時点で全部入りにしなくてもよいが、以下は最初から入れる価値が高い。

- `model-catalog.yaml` と `compression-profiles.yaml` の分離
- `Model Adapter Interface` の導入
- `standard / code_focused / lightweight` の profile 別名
- 保存済み評価セットによる比較コマンド
- profile 単位の fallback 指定

これだけあれば、初期実装が `Qwen3-1.7B` 中心でも、後から `Qwen2.5-Coder-1.5B-Instruct` や別の軽量モデルへ寄せやすい。

## 24. MCP活用方針

### 24.1 方針

本プロダクトでは、MCP を圧縮エンジンそのものとして使うのではなく、
Codex などのホストアプリから安全に呼び出すための連携インターフェースとして使う。

役割分担:

- Core: ローカルLLMによる圧縮処理
- CLI: 直接実行用の入口
- MCP Server: Codex 連携用の入口
- Local App: 将来のGUIと設定管理

この分離により、圧縮ロジックは CLI と MCP の両方から再利用できる。

### 24.2 MVPにおけるMCPの位置づけ

MVP では、Codex からこの圧縮機能を呼び出せるようにするため、ローカル MCP Server を提供する。
ただし、MCP を使うのはあくまで「手動または明示的な実行」に限定し、自動送信や暗黙圧縮は行わない。

MVPでの前提:

- ローカル動作のみ
- APIキー不要
- クラウドLLMへの自動送信なし
- 圧縮前後の内容をユーザーが確認できる
- Codex から明示的に tool を呼ぶ

### 24.3 MVPの推奨transport

MVP では `stdio` を第一選択とする。

理由:

- ローカルPC内で完結しやすい
- 外部到達可能なHTTPポートを開かずに済む
- APIキー不要・非プロキシ方針と整合する
- Codex などのローカルホスト連携と相性がよい

`Streamable HTTP` は将来対応候補とし、MVPでは対象外とする。

### 24.4 MVPで公開するMCP tools

MVPでは、まず最小限の4〜5個の tools に絞る。

必須 tools:

1. `compress_prompt`
   - 生の入力文を受け取り、圧縮後プロンプトを返す
   - mode, level, profile を指定できる

2. `estimate_tokens`
   - 入力文または圧縮結果に対する推定トークン数を返す
   - モデルまたは profile ごとの推定に対応する

3. `list_profiles`
   - 利用可能な profile 一覧を返す
   - `standard`, `code_focused`, `lightweight` などを返す

4. `compare_profiles`
   - 同一入力に対して複数 profile の結果を並べて返す
   - 圧縮率、保持条件、リスク、応答時間を比較できる

任意追加:

5. `get_profile_details`
   - 指定 profile が参照する model, policy, fallback を返す

### 24.5 MVPではまだ公開しないMCP tools

初期段階では、広げすぎると評価と保守が重くなるため、以下はMVP外とする。

- 自動ベンチ実行ツール
- モデルダウンロード管理ツール
- 量子化変換ツール
- クリップボード操作ツール
- 任意Web UIへの貼り付けツール
- クラウドLLM送信ツール

### 24.6 将来候補のMCP prompts

MVP直後の拡張として、Codex用途に合わせた prompt templates を公開できる。

候補:

- `codex_bugfix_compress`
- `codex_refactor_compress`
- `codex_log_analysis_compress`
- `codex_review_request_compress`

これらは user-controlled な prompt として扱い、ユーザーが明示的に選ぶ形にする。

### 24.7 将来候補のMCP resources

MCP resources は、モデル選定や比較結果の可視化に役立つ。

候補 resources:

- `resource://profiles`
- `resource://models`
- `resource://evals/latest`
- `resource://evals/history`
- `resource://compression-policies/codex_optimized`

読み取り専用を基本とし、MVPでは resource の更新操作は提供しない。

### 24.8 tool入出力の共通方針

MCP tool の入出力は、CLIの共通JSONと揃える。
これにより、CLI 実装と MCP 実装でロジックを共通化しやすくなる。

入力の基本項目:

- input_text
- task_type
- compression_mode
- compression_level
- profile
- constraints

出力の基本項目:

- distilled_prompt
- preserved_requirements
- removed_content_summary
- risk_flags
- should_send_original
- metrics

### 24.9 セキュリティ方針

MCP連携でも、もともとの安全方針は維持する。

必須方針:

- APIキーを保持しない
- 外部クラウドへの自動送信を行わない
- 機微入力をそのまま外部へ転送しない
- tool 実行前にユーザーが内容を確認できる前提を保つ
- roots で許可された範囲だけを参照する
- ログには全文を不用意に残さない

### 24.10 rootsの使いどころ

CodexからMCP serverを使う場合、roots を活かして参照可能なワークスペース境界を明確にする。

使いどころ:

- 対象ファイルを含むプロジェクトだけを参照対象にする
- 複数リポジトリ利用時に対象repoだけへ限定する
- 不要な個人ファイル、別案件ファイルの混入を避ける

### 24.11 推奨MVP構成

MVPとしては、以下の構成がもっとも筋がよい。

1. Core圧縮エンジン
2. CLI
3. `stdio` ベースのローカル MCP Server
4. `compress_prompt` / `estimate_tokens` / `list_profiles` / `compare_profiles`

この順で入れると、まず CLI で品質を詰め、その後に Codex 連携を足しやすい。

### 24.12 期待できるメリット

MCPを活用することで、以下の利点がある。

- Codexから直接呼び出せる
- CLIとMCPで同じコア処理を使い回せる
- profile や model の比較結果を構造化して返せる
- 将来、prompts や resources に自然に拡張できる
- セキュリティ方針を保ったままローカル統合できる

### 24.13 注意点

MCPを採用しても、これだけで価値が出るわけではない。
本質的な価値は、圧縮後もCodexの回答品質が落ちないことにある。

そのため、MCP対応は重要だが、優先順位としては以下を守る。

1. 圧縮品質
2. CPUでの実用速度
3. CLIでの実用性
4. MCPによるCodex統合
